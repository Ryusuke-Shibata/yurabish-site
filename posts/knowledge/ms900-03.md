今回はセキュリティ機能の中でも、Microsoft 365を利用するための入口になる認証周りを整理します。

Microsoft 365はクラウドサービスなので、インターネット環境があればどこからでも利用できます。
便利な反面、適切な制御をしないと、誰でもアクセスできてしまう状態にもなります。

そのため、

- 誰が利用するのか
- 本当に本人なのか
- どの条件なら利用を許可するのか

こういった考え方が重要になります。

今回はMicrosoft Entra IDと、認証・アクセス制御の仕組みを簡単に整理します。

## Microsoft Entra ID

Microsoft Entra IDは、Microsoft 365における認証基盤です。

主な役割は以下の4つです。

- ユーザー管理
- グループ管理
- 認証
- アクセス管理

以前は「Azure Active Directory（Azure AD）」という名前でしたが、現在はMicrosoft Entra IDへ名称変更されています。
変更されてそれなりに時間が経っている気がしますが、たまに旧表記で記載されていますね。
Microsoft 365の利用者は普段あまり意識しませんが、裏側では多くのサービスがここを利用しています。

## シングルサインオン

Microsoft Entra IDにはシングルサインオン（SSO）機能があります。

これは、

> 1つの認証情報で複数のサービスを利用できる仕組み

です。

例えば、Microsoft 365の各サービスやその他SaaSを利用する場合でも、毎回認証を要求されるわけではありません。

利用者の負担軽減だけではなく、

- パスワード管理の簡略化
- セキュリティ向上

という効果もあります。

## MFA

MFA（多要素認証）は、複数の認証要素を利用して本人確認を行う仕組みです。

認証要素は主に以下の3種類です。

- 知識情報（パスワードなど）
- 所持情報（スマートフォンなど）
- 生体情報（顔認証・指紋認証など）

比較的よく見る例としては、**パスワード入力＋スマートフォン認証**があります。
最悪パスワードが漏洩しても、スマートフォンがなければサインインできないので、情報漏洩時のリスクを大きく減らせます。

## 条件付きアクセス

MFAが「本人確認」なら、条件付きアクセスは「利用条件の制御」です。

例えば、以下のように設定すると、

- 社内からのみ許可
- 管理済み端末のみ許可
- 特定地域以外はブロック
- 社外接続時だけMFA必須

本人であっても、危険な環境からのアクセスは制限できるというわけです。

セキュリティと利便性は多くの場合トレードオフですが、条件付きアクセスはそのバランスを調整するための機能と言えるでしょう。

## まとめ

- Microsoft Entra IDはMicrosoft 365の認証基盤
- MFAは本人確認を強化する仕組み
- 条件付きアクセスは利用条件を制御する仕組み

Microsoft 365を利用する上では、サービスそのものだけではなく「どう安全に利用するか」も重要になります。
そして、MFAや条件付きアクセスは、安全なテナントを構築するための重要な要素の一つです。
